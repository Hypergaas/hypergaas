# Decision: v0.1 build stack — pnpm + vitest + TC39 stage-3 decorators + ts-morph

**Status**: Accepted — 2026-05-15 via batched `approve`-tier queue item
(`inbox/approved.jsonl` line 2, resolved 2026-05-15T18:43:07Z).

**Scope**: The four build-tooling choices that jointly define what the v0.1 `sdk/`
build looks like. Locked together as one batch per
`meta/skills/batch-related-approvals.md`. Re-opening any individual choice is an
`approve`-tier amendment per `meta/skills/react-cleanly-to-batched-approval.md`,
not a unilateral relaxation.

**Path convention**: This ADR lives at `sdk/docs/decisions/<topic>.md`. The
convention mirrors the agent-company's own `domains/engineering/decisions/<topic>.md`
(see `role-registry.md`), so decision-doc semantics are consistent across the two
repos the agent company touches. Future decision records in `sdk/` go in this
directory under a topic-keyed filename. We are not using a numbered `adr/NNNN-*.md`
convention; that choice is documented here so it does not get re-decided.

## Context

On 2026-05-15 (day-0 ship session), four build-tooling questions surfaced that
jointly define the shape of the v0.1 build: package manager, test framework,
decorator mode, and reflection mechanism for schema codegen. The four are
*jointly load-bearing* — the disqualifying logic for `reflect-metadata` only
makes sense once the TC39 stage-3 decision is on the table; the choice of
package manager constrains how the ts-morph codegen CLI is later invoked; the
test framework choice interacts with the decorator-mode choice via TS strict-
mode + ESM expectations.

Per `meta/skills/batch-related-approvals.md`, items that share a fate must be
filed as one queue item so the operator decides the whole stack in one read,
rather than approving piecemeal and creating re-litigation when later items
contradict earlier ones. The four were filed as a single `approve`-tier item
(`inbox/approved.jsonl` line 2) and approved as a batch the same day.

This ADR pins those four choices in source so the consumer-side discipline of
`meta/skills/react-cleanly-to-batched-approval.md` has a canonical, written-down
referent: when a future ship session asks "what did we pick for X?", the answer
is here, not a re-derivation from the queue log. Re-opening any of the four is
an explicit `approve`-tier amendment.

## Decision

Each subsection below pins one of the four batched options. Lean, alternatives,
and disqualifying reason are quoted from `inbox/approved.jsonl` line 2 `details`
verbatim — not re-derived.

### Package manager — **pnpm**

> Package manager: pnpm vs npm vs bun. Lean: pnpm. Trade: pnpm gives workspace
> support (we'll want it once examples/ + react-integration packages exist), is
> the de-facto standard for the Vercel AI SDK reference, and has the fastest
> cold install. npm is most universal but slower and weaker on workspaces. bun
> is fastest of all but is still maturing on TS-decorator + ESM edge cases —
> risky for v0.1.

Source locations in `sdk/`:
- `package.json` → `"packageManager": "pnpm@10.17.1"` (set day-1, commit
  `f9fdcd6`)
- `pnpm-lock.yaml` committed (the lockfile is the ground truth; npm/yarn
  lockfiles must not be added)

### Test framework — **vitest**

> Test framework: vitest vs node:test vs jest. Lean: vitest. Trade: vitest is
> TS-native, ESM-native, has the best TS-strict ergonomics, runs in-process
> with HMR, and is what most reference TS-SDK projects use. node:test ships
> with Node and has zero dependencies but lacks the matcher ergonomics and
> watch UX; reasonable fallback if 'minimize deps' becomes a higher-order
> goal. jest is mature but ESM/TS friction is real and the project is in
> maintenance mode.

Source locations in `sdk/`:
- `package.json` → `vitest` in `devDependencies`, `test` and `test:watch`
  scripts
- `vitest.config.ts` → `include: ["src/**/*.test.ts"]`, node environment
- `tsconfig.test.json` → editor / standalone typecheck for test files without
  bleeding test types into `dist/`

### Decorator mode — **TC39 stage-3 decorators** (Node 22+ / TS 5.0+)

> Decorator mode: TC39 stage-3 vs experimentalDecorators. Lean: TC39 stage-3.
> Trade: TC39 is the future, matches strict-mode philosophy
> (`ClassMethodDecoratorContext` is well-typed), works on Node 22+ / TS 5.0+,
> and is the only mode compatible with the registry-aware decorator signature
> in the spec (§1). experimentalDecorators is broader-supported but ties us
> to reflect-metadata semantics that erase generics — disqualifying for our
> schema-derivation use case (spec §3.a path B).

Source locations in `sdk/`:
- `tsconfig.json` → `experimentalDecorators: false` (decorators flag for
  legacy mode left explicitly off; TC39 stage-3 is the default in TS 5.0+
  and needs no opt-in flag)
- `package.json` → `engines.node >= 22`
- `src/runtime/action-registry/agent-action.ts` →
  `ClassMethodDecoratorContext`-typed decorator factory
  (`createAgentActionDecorator(registry)`), uses `addInitializer` for
  per-instance registration; rejects static/private methods and non-method
  decoration kinds at decoration time

### Reflection mechanism — **ts-morph build-time codegen**

> Reflection mechanism: ts-morph build-time codegen vs reflect-metadata
> runtime. Lean: ts-morph. Trade: reflect-metadata is disqualified by (3) —
> it can't preserve generics. Manual zod schemas (a third option) are
> rejected because they force parallel definition (TS type + schema), which
> is exactly the schema-drift the action registry exists to eliminate
> (BRIEF.md). ts-morph is the only path that preserves the developer's
> literal TS types in the emitted tool schema while keeping the public
> surface type-safe. Tension to flag: ts-morph requires literal-array
> audienceRoles for clean schema emission; computed audiences degrade. Spec
> §3.c proposes a strict emitter (computed audiences raise build error).
> Surfaced separately as notify-tier.

Source locations in `sdk/`:
- Spec: `sdk/docs/specs/action-registry.md` §3 (full Path A / B / C
  comparison) and §3.b (recommendation rationale)
- Implementation: **deferred.** No ts-morph code on disk today. The runtime
  registry has a permissive `Record<string, unknown>` schema fallback with a
  one-time `console.warn` per spec §3.b — see
  `sdk/src/runtime/action-registry/registry.ts`. The ts-morph CLI
  (`gaasdk extract` or equivalent) lands at the v0.1 public-type-surface
  commit, not before.

## Strict-emitter rule for ts-morph schema codegen

First-class rule, not a footnote. Source of authority: `inbox/approved.jsonl`
line 3 (`notify`-tier, resolved 2026-05-15T18:44:16Z) plus spec §3.c.

**Rule**: `audienceRoles` must be a **literal array of role keys** at the
decoration site.

- Literal arrays (`audienceRoles: ['owner', 'dispatcher']`) resolve cleanly.
- Inline `const` references to a typed array (e.g.,
  `const dispatcherOrOwner: ForgeProRole[] = ['dispatcher', 'owner']`) may be
  followed by ts-morph if in scope and statically resolvable. The emitter does
  this on a best-effort basis; if the reference cannot be resolved without
  executing the developer's code, it fails the same way a computed expression
  fails.
- **Computed expressions are a hard build error.** Examples that fail:
  `audienceRoles: getAudienceFor("schedule:read")`,
  `audienceRoles: [...baseAudience, "owner"]` where `baseAudience` is imported
  from another package, conditional spreads, function-call results.
- The build error message must be specific: `"audienceRoles must be a literal
  array of role keys; compute audiences via audience.self(...) predicates
  instead, or inline the literal array"`.
- **Predicate audiences via `audience.self(...)` remain fully supported** as
  entries inside the literal array. The rule is about the *array literal
  shape*, not about what entries the array contains. `audience.self((ctx,
  params) => ...)` is a typed predicate constructor (per
  `domains/engineering/decisions/role-registry.md`) and the ts-morph emitter
  treats it as a black-box entry — the runtime predicate is what evaluates;
  the emitter doesn't need to introspect its body.

**Why strict rather than lenient (emit-wide-and-warn)**: lenient mode silently
degrades the typo-prevention guarantee that is the entire point of the
registry. The lenient option was filed and explicitly rejected on 2026-05-15;
the rejection is locked in by the resolved-approved `notify` and not
re-litigated here.

This rule must be documented in the public docs at the moment the v0.1 public
type surface ships — flagged here so it is not a surprise at that future
`approve`-tier moment.

Cross-reference: spec §3.c; `inbox/approved.jsonl` line 3.

### Open gaps surfaced as notify

Per the slice directive, gaps that the staging table cannot answer without an
operator decision are surfaced as `notify`s rather than silently invented in
the ADR.

1. **`audience.self()` in v0.1 vs. v0.2?** `domains/engineering/decisions/role-registry.md`
   names `audience.self(...)` as a v0.1 public export. The staging table above
   reflects that. **Lean: ship `audience.self` in v0.1** — it is the only
   non-`role-key` audience entry the PoC actually exercises, and removing it
   would force the canonical `JobService.getTechSchedule` worked example to
   change shape. Locked decision per `role-registry.md`; no notify needed.
2. **`AudiencePredicate<P>` exported, or kept internal and only-usable-via
   `audience.self`?** Today the staging table lists it as exported so
   developers can type their own predicate factories. If the operator wants
   to keep the predicate factory surface minimal at v0.1 (only `audience.self`,
   no developer-defined predicate constructors), this row narrows. Surfaced as
   `notify` (see `inbox/pending.jsonl`).
3. **`pnpm run codegen` wired into `pnpm run build`, or invoked as a separate
   step in CI?** The staging table lean is "wired into `build` (codegen runs
   before `tsc`)". Alternative is a separate `pnpm run codegen && pnpm run
   build` invocation. The trade is "developer ergonomics (one command) vs.
   build observability (codegen failures distinct from typecheck failures)".
   Surfaced as `notify` (see `inbox/pending.jsonl`).

No other gaps surfaced. The remaining rows are pinned by prior approvals or
existing source on disk.

## Consequences

**Lock-in cost per choice**:

- **pnpm**: small. The lockfile commits us to pnpm's resolution semantics; a
  future switch to npm/yarn/bun would require regenerating the lockfile and
  re-testing the install graph. No code change. Re-opening this is an
  `approve`-tier amendment per `react-cleanly-to-batched-approval` step 4.
- **vitest**: small. The test surface is `describe`/`it`/`expect` —
  framework-portable patterns. A switch to node:test would require rewriting
  test setup + matcher calls but not test logic. Re-opening is `approve`-tier
  amendment.
- **TC39 stage-3 decorators**: medium. The decorator factory
  (`agent-action.ts`) uses `ClassMethodDecoratorContext` and `addInitializer`,
  both stage-3-specific. A switch to `experimentalDecorators` would force
  reflect-metadata which erases generics (spec §3.a path B) and is therefore
  structurally disqualified by the registry-aware decorator signature in
  spec §1. **Effective lock-in is high** — re-opening this would require
  re-opening the reflection-mechanism choice as well, because the two are
  joint. This is exactly why the four were batched.
- **ts-morph**: medium. No code on disk depends on ts-morph today; the v0.1
  public-type-surface commit is what binds us. Re-opening before that commit
  is cheap; re-opening after that commit means a build-step rewrite.

**What re-opening looks like**: file a new `approve`-tier item per
`meta/skills/react-cleanly-to-batched-approval.md` step 4 with the
disqualifying reason (not "suboptimal" — "structurally incompatible with hard
invariant X"). The amendment goes through the queue; no worker unilaterally
relaxes the lock. If a *forcing function* surfaces during implementation that
makes a locked option unworkable, file a `notify` per
`react-cleanly-to-batched-approval` step 5 and stop the in-flight session —
the coordinator's next dispatch decides whether to amend or work around.

**The strict-emitter rule** locks in `audienceRoles` as a literal array at the
decoration site, full stop. The public docs at v0.1-launch must surface this
prominently so developers do not discover it via a confusing build error.
